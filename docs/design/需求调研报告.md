## 1. 多图 OCR + 文档合成

### 需求描述
- 输入：飞书传图、文档内图片、截图、扫描 PDF 等
- 输出：可编辑 Markdown / 结构化文档
- 核心能力：多图批量处理、表格识别、公式识别、LLM 校错

### 现有方案对比

| 方案 | 类型 | 表格识别 | 公式识别 | 多语言 | 批量处理 | 准确率 | 速度 | 成本 |
|---|---|---|---|---|---|---|---|---|
| **PaddleOCR 3.0** | 开源 VLM | ✅ PP-StructureV3 | ✅ LaTeX | 111 语言 | ✅ 批量+API | 🏆 OmniDocBench 94.5% | 快（0.9B 轻量模型） | 免费开源 |
| **Tesseract 5** | 开源 OCR | ❌ 需外挂 | ❌ | 100+ 语言 | ✅ CLI 批量 | 中（~85-90%，对干净文本好） | 快 | 免费开源 |
| **MinerU 2.7** | 开源文档解析 | ✅ HTML 格式 | ✅ LaTeX | 109 语言 | ✅ CLI/Gradio | 高（pipeline 82+, VLM 90+） | 中（需 GPU） | 免费开源 |
| **DeepSeek-OCR** | 开源 VLM | ✅ | ✅ | 多语言 | ✅ vLLM 推理 | 高（研究级） | 中（需 A100 级 GPU） | 免费开源 |
| **Google Vision API** | 云 API | ✅ | ❌ 基础 | 多语言 | ✅ 批量 API | 高 | 快 | $1.5/千次 |
| **Azure Computer Vision** | 云 API | ✅ | ❌ 基础 | 多语言 | ✅ 批量 API | 高 | 快 | $1/千次（首 100 万） |

### 关键发现

**1. PaddleOCR 3.0 是当前开源 OCR 的标杆**
- PaddleOCR-VL-1.5 (0.9B 参数) 在 OmniDocBench v1.5 上达到 94.5% 精度，超越商业方案
- 支持 PP-StructureV3 复杂文档解析（表格→HTML、公式→LaTeX）
- 支持 MCP Server，可直接与 Agent 集成
- 已被 MinerU、RAGFlow 等项目深度集成
- GPU 不是必须——支持纯 CPU 运行（精度稍低）

**2. MinerU = 专业文档解析利器**
- 诞生于 InternLM 预训练阶段，专为科学文献设计
- 默认引擎从 pipeline 切换到 hybrid（VLM + pipeline 混合），精度 90+
- 核心优势：自动去除页眉页脚、自动检测扫描 PDF 并启用 OCR、阅读顺序恢复
- 支持跨页表格合并
- 缺点：依赖较多（需要 Python 环境、GPU 推荐），安装复杂

**3. DeepSeek-OCR 需要高端 GPU**
- 2025/10 发布，2026/1 已有 DeepSeek-OCR2
- 设计理念：从 LLM 视角探索视觉编码器的压缩边界
- 需要 A100-40G，concurrency ~2500 tokens/s
- 适合有 GPU 资源的场景，**当前 Boss 环境暂不可用**

**4. 已有测试数据佐证（2026-03-15 测试）**
- 在 NVIDIA 免费端点测试 18 个模型，llama-4-maverick 效果最佳（11s，准确率高）
- 云 API 方案适合无 GPU 场景下的 OCR 替代

### LLM 校错方案

| 步骤 | 方法 |
|---|---|
| OCR 原始输出 | PaddleOCR/MinerU 输出结构化文本 |
| LLM 校错 | 将 OCR 结果 + 原图发送给视觉大模型，让模型对照修正 |
| Prompt 设计 | "请对照图片，修正以下 OCR 文本中的错误，保留原格式和结构" |
| 推荐模型 | qwen3.5-397b / llama-4-maverick（已有测试验证）|

### 🏆 推荐方案

**主选：PaddleOCR 3.0 + LLM 校错**
- **理由：**
  1. OmniDocBench 94.5% 精度，表格/公式/复杂版面全面领先
  2. 支持纯 CPU 运行，不依赖 GPU
  3. PP-StructureV3 直接输出 Markdown/JSON，与知识库格式天然契合
  4. MCP Server 可与 OpenClaw 集成
  5. 完全免费开源，无 API 调用成本
- **备选：MinerU** — 处理 PDF 优先级文档时作为补充
- **校错：** 调用 OpenRouter API（qwen3.5 / llama-4-maverick）进行二次校验

## 2. 文章链接 → 笔记提炼

### 需求描述
- 输入：微信公众号、CSDN、博客等链接
- 输出：去水货留干货的 Markdown 笔记
- 关键：保留有价值细节链接，不是简单 copy，是提炼

### 现有方案对比

| 方案 | 类型 | 内容提取 | AI 提炼 | 微信支持 | 价格 | 笔记输出 |
|---|---|---|---|---|---|---|
| **简悦 (SimpRead)** | Chrome 扩展 | ✅ 阅读模式 | ❌ 需配合 | ✅ 适配 | ¥88 买断 | Markdown/Notion/Obsidian |
| **Cubox** | 全平台 SaaS | ✅ AI 阅读助手 | ✅ AI 摘要 | ✅ 小程序 | ¥128/年 | 内置阅读器 |
| **Readwise Reader** | 全平台 SaaS | ✅ 阅读模式 | ✅ Ghostreader AI | ❌ 需绕路 | $9.9/月 | Markdown 导出/Obsidian 同步 |
| **Markdownload** | 浏览器扩展 | ✅ Readability | ❌ | ✅ 浏览器内 | 免费开源 | 纯 Markdown |
| **自建方案** | CLI + Agent | ✅ web_fetch | ✅ LLM 提炼 | ⚠️ 需处理 | API 成本 | 完全自定义 |

### 关键发现

**1. Omnivore 已停运**
- 2024 年被 ElevenLabs 收购后停止服务
- 曾是开源稍后阅读的最佳选择，现已不可用

**2. 简悦 (SimpRead) — 中文生态最强**
- Chrome/Firefox 扩展，阅读模式体验极佳
- 支持微信公众号、知乎、CSDN 等中文网站适配
- 可导出 Markdown/Notion/Obsidian/语雀等
- 缺点：无 AI 提炼能力，需要配合其他工具

**3. Cubox — AI 阅读助手定位**
- 内置 AI 摘要功能
- 支持微信小程序收藏
- 适配国内生态
- 缺点：订阅制（¥128/年），数据在 Cubox 云端

**4. Readwise Reader — 英文生态最强**
- Ghostreader AI 可做笔记提炼
- 与 Kindle/Instapaper/iBooks 等无缝同步
- 缺点：中文支持一般，微信公众号需绕路，价格较贵

**5. Markdownload — 最简单的 Markdown 导出**
- 基于 Readability.js + Turndown，质量不错
- 支持 Obsidian 直接集成
- 缺点：无 AI 能力，对复杂页面适配有限

### 微信公众号反爬策略

| 策略 | 说明 | 实现方式 |
|---|---|---|
| 浏览器自动化 | 使用已登录的浏览器访问 | Playwright/Puppeteer + 已有 cookie |
| Readability 解析 | 清理广告和冗余元素 | Mozilla Readability.js |
| User-Agent 伪装 | 模拟正常浏览器 | 自定义 UA header |
| 代理池 | 避免 IP 限制 | 旋转代理 |
| 微信内置浏览器 | 通过微信转发获取 | 需手机端配合 |

### "去水货留干货" LLM 提炼 Prompt 设计

```markdown
你是一个专业的知识提炼专家。请对以下文章进行深度提炼：

## 提炼规则：
1. **保留核心论点和关键数据**（百分比、案例、数据来源）
2. **去除废话**：删除重复论述、营销语、过渡性废话、作者自我介绍
3. **保留有价值链接**：原文中的参考资料、工具链接、数据来源
4. **结构化输出**：按逻辑层次组织，使用标题/列表
5. **补充理解**：如原文有模糊表述，可补充 1-2 句背景说明

## 输出格式：
# [标题]
> 原文：{url} | 提炼日期：{date}

## 核心要点
- [3-5 个关键点]

## 详细内容
[结构化提炼]

## 参考链接
- [有价值的外部链接]
```

### 🏆 推荐方案

**主选：自建 Web Fetch + LLM 提炼管线**
- **理由：**
  1. 完全控制"去水货"逻辑，可针对中文网站优化
  2. `web_fetch` 已内置（OpenClaw），基于 Readability
  3. 微信公众号可通过浏览器自动化（Playwright）解决
  4. LLM 提炼 prompt 可迭代优化，比固定规则灵活得多
  5. 成本可控：用 DeepSeek API 约 ¥0.001/篇
- **辅助工具：** 简悦作浏览器端补充（手动收藏场景）

## 3. 多媒体链接 → 学习笔记

### 需求描述
- 输入：抖音、B站、YouTube 视频链接
- 输出：结构化学习笔记（字幕提取 + LLM 总结）

### 现有方案对比

| 方案 | YouTube | B站 | 抖音 | 字幕提取 | LLM 总结 | 价格 |
|---|---|---|---|---|---|---|
| **yt-dlp** | ✅ | ✅ BiliBili 系列 | ✅ | ✅ --write-subs | ❌ 需配合 | 免费开源 |
| **NoteGPT** | ✅ | ✅ | ✅ | ✅ 自动 | ✅ AI 摘要 | $9.99/月 |
| **Harvest** | ✅ | ❌ | ❌ | ✅ 自动 | ✅ AI 笔记 | 免费 |
| **八爪鱼/后裔采集** | ⚠️ | ⚠️ | ⚠️ | ❌ | ❌ | ¥99+/月 |
| **自建方案** | ✅ | ✅ | ✅ | ✅ yt-dlp | ✅ LLM | API 成本 |

### 关键发现

**1. yt-dlp 是视频字幕提取的事实标准**
- 支持 1000+ 站点，包括 BiliBili（BiliBiliAudio/BiliBiliBangumi/BilibiliCheese 等多提取器）
- 支持抖音
- 字幕提取命令：`yt-dlp --write-subs --sub-lang zh --skip-download <url>`
- 已在 Boss 环境安装

**2. B站字幕提取特殊处理**
- B站字幕为 JSON 格式（非 SRT），yt-dlp 自动处理
- 需登录才能获取部分字幕（大会员内容）
- 可通过 `--cookies-from-browser` 获取登录态

**3. 抖音提取策略**
- yt-dlp 支持抖音，但需要处理反爬
- 字幕通过 ASR 生成（抖音自动生成字幕）
- 备选：通过浏览器自动化提取

**4. LLM 总结视频内容最佳实践**

```markdown
## Prompt 模板
你是一个学习笔记专家。请根据以下视频字幕，生成结构化学习笔记：

### 输出要求：
1. **摘要**：3 句话说清视频核心内容
2. **知识框架**：用层级列表梳理知识结构
3. **关键概念**：列出核心概念及简短解释
4. **实操要点**：如有操作步骤，提取为 checklist
5. **思考题**：基于内容提出 2-3 个延伸思考
6. **时间戳索引**：重要内容标注原始时间戳（如果字幕包含）

### 注意：
- 保留原始数据和引述
- 去除口语化表达（"嗯"、"那个"、"就是说"等）
- 标注可能不准确的地方（字幕 ASR 错误）
```

**5. 现成工具评估**
- **NoteGPT**：功能最全（YouTube/B站/抖音 + AI 摘要），但 $9.99/月 且数据在云端
- **Harvest**：免费但仅支持 YouTube
- **结论**：没有完美现成工具，自建方案更有优势

### 🏆 推荐方案

**主选：yt-dlp + LLM 总结管线**
- **理由：**
  1. yt-dlp 已安装，支持 YouTube/B站/抖音
  2. 字幕提取一行命令搞定
  3. LLM 总结 prompt 可针对不同内容类型（教学/评测/新闻）定制
  4. 完全控制数据，不依赖第三方服务
  5. 成本极低：字幕提取免费，LLM 调用约 ¥0.01-0.05/视频
- **实现流程：**
  ```
  URL → yt-dlp 字幕提取 → 格式化（SRT→纯文本）→ LLM 总结 → Markdown 笔记
  ```

## 4. 与 AI Agent 集成

### 需求描述
- CLI 工具，与 OpenClaw 集成
- 可选：API、Web UI
- 作为 OpenClaw skill 提供

### 现有方案对比

| 集成方式 | 实现复杂度 | 灵活性 | 与 OpenClaw 契合度 | 推荐场景 |
|---|---|---|---|---|
| **OpenClaw Skill (SKILL.md)** | 低 | 高 | 🏆 最佳 | CLI 工具封装，通过 exec 调用 |
| **MCP Server** | 中 | 高 | ✅ 良好 | 需要被多 Agent 调用 |
| **REST API + CLI 前端** | 中 | 最高 | ✅ 良好 | 需要 Web UI 或外部调用 |
| **纯 CLI 工具** | 低 | 中 | ✅ 良好 | 简单脚本化使用 |
| **飞书 Bot 集成** | 中 | 中 | ✅ 特定场景 | Boss 通过飞书发送链接 |

### 关键发现

**1. OpenClaw Skill 体系**
- OpenClaw 已有丰富的 skill 生态：Obsidian、Notion、GitHub、nano-pdf 等
- Skill 结构：`SKILL.md`（描述）+ 脚本（实现）
- 通过 `exec` 工具调用 CLI，通过 `web_fetch` 提取网页
- 支持 cron job 定时执行、heartbeat 主动检查

**2. 已有可复用的 OpenClaw 能力**
- `web_fetch`：网页内容提取（基于 Readability）
- `browser`：浏览器自动化（解决反爬）
- `feishu_doc`：飞书文档读写
- `feishu_bitable`：多维表格（任务追踪）
- `nano-pdf`：PDF 编辑
- `obsidian` skill：Obsidian 笔记操作
- `summarize` skill：URL/PDF 摘要

**3. PaddleOCR MCP Server**
- PaddleOCR 3.0 已提供 MCP Server
- 可直接与 Claude Desktop / OpenClaw Agent 集成
- 这意味着 OCR 能力可以作为 MCP 工具暴露给 Agent

**4. CLI 工具设计最佳实践**

```python
# 推荐 CLI 结构：kb-tool
kb-tool
├── ocr     # 多图 OCR → Markdown
│   ├── --input images/     # 输入图片目录
│   ├── --output doc.md     # 输出文件
│   ├── --llm-correct       # 启用 LLM 校错
│   └── --format md|json    # 输出格式
├── article # 文章链接 → 笔记
│   ├── --url <url>         # 文章链接
│   ├── --output note.md    # 输出文件
│   ├── --depth deep|brief  # 提炼深度
│   └── --wechat            # 微信特殊处理
├── video   # 视频链接 → 笔记
│   ├── --url <url>         # 视频链接
│   ├── --lang zh|en        # 字幕语言
│   └── --format md|json    # 输出格式
└── sync    # 笔记同步
    ├── --to feishu          # 同步到飞书
    ├── --to obsidian        # 同步到 Obsidian
    └── --to notion          # 同步到 Notion
```

**5. 飞书集成路径**
- 直接发送图片 → OpenClaw 接收 → OCR → 写入飞书文档
- 发送链接 → OpenClaw 提取 → 写入飞书知识库
- 利用飞书多维表格做任务队列

### 🏆 推荐方案

**主选：OpenClaw Skill + CLI 工具**
- **理由：**
  1. 与 Boss 已有工作流无缝集成（飞书对话 → Agent 触发）
  2. 可复用 OpenClaw 的 `web_fetch`、`browser`、`feishu_doc` 等能力
  3. Skill 机制天然支持扩展和迭代
  4. CLI 工具可独立使用，也通过 exec 被 Agent 调用
  5. PaddleOCR MCP Server 可直接集成
- **架构：**
  ```
  用户输入（飞书/CLI）
      ↓
  OpenClaw Agent（调度层）
      ↓
  kb-tool CLI（核心逻辑）
      ├── PaddleOCR / MinerU（OCR）
      ├── web_fetch + 浏览器（内容提取）
      ├── yt-dlp（字幕提取）
      └── LLM API（提炼/校错/总结）
      ↓
  输出（Markdown → 飞书/Obsidian/本地）
  ```
