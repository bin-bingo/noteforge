## 讨论 3：ACP + OpenCode 配置（14:24 - 15:42）

### 讨论内容

**Boss 提问：** 为什么 ACP 超时？

**灵虾排查结果：**
- 原因 1：**任务太大** —— 单次提交的编程任务粒度过粗
- 原因 2：**模型冷启动慢** —— 部分模型首次响应时间过长

**模型切换过程：**

| 模型 | 表现 | 结论 |
|------|------|------|
| qwen3-coder | 太慢 | ❌ 淘汰 |
| deepseek-v3.2 | 慢 | ❌ 淘汰 |
| mistral-small | 快但能力弱 | ❌ 淘汰 |
| kimi-k2.5 | 速度和能力平衡 | ✅ 暂定 |

**Boss 纠正：** 等大模型就好，不用换小模型。

**关键认知：**
- **ACP 适合 IDE 集成场景**（交互式编程、实时反馈）
- **ACP 不适合批量编程任务**（大任务容易超时）
- **正确方式：** sub-agent 用 `opencode run`，给够超时时间

### 决策
- ACP 定位为 IDE 集成工具，不用于批量任务
- 批量编程任务使用 `opencode run` + 充足超时
- 模型选择：等大模型响应，不降级到小模型

### 待办
- [ ] 确认 kimi-k2.5 作为 ACP 默认模型
- [ ] 文档化 ACP vs opencode run 的使用场景划分
- [ ] 优化超时配置策略
