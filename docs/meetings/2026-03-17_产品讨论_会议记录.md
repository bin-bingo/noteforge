### 议题 1：提炼策略——"去水留干"有没有问题？

**Boss 提问：** 我们的提炼，有没有丢信息密度的问题？或者我们的提炼，是不是用户想要的？

**灵虾分析：**

这两个问题其实是同一个问题的两面。

**1. 信息密度丢失——肯定丢。问题是丢的恰好是不是用户不在乎的？**

传统 LLM 提炼会丢失：
- **论证过程** —— 只留结论，丢了为什么这么想
- **语气和立场** —— 作者的判断力被扁平化成"要点"
- **上下文衔接** —— "这个跟那个的关系"变成独立的 bullet
- **细节数据** —— 数字、引用可能被当作"次要信息"删掉

**结果：** 一篇 5000 字的深度分析 → 500 字的要点列表。看起来"去水"了，但读完要点你其实没理解作者到底在说什么。

**2. 提炼 = 用户想要的吗？不一定。分两种情况：**

用户确实想要提炼的场景：
- 快速判断"这篇文章值不值得细读"
- 事后回顾"那篇文章讲了什么来着"
- 知道自己不会读原文，至少留个精华版

用户其实不想要提炼的场景：
- 技术文档 —— 删掉任何细节都可能致命
- 深度分析文章 —— 论证过程就是价值
- 研究论文 —— 方法论和数据不能省
- 自己珍藏的"以后要精读"的内容

**这引出一个产品哲学问题：**

| 路线 | 代表 | 做什么 | 用户体感 |
|------|------|--------|---------|
| **提炼派** | Mem.ai, Cubox | "帮你总结好了" | 省时间，但信息变薄 |
| **归档派** | 简悦, Pocket | "帮你存好了原文" | 信息完整，但还得自己看 |
| **??** | 我们？ | ？ | ？ |

**灵虾建议：v1 不做"去水留干"，做"原文+AI 旁注"。**

不是替用户"删"，而是帮用户"标"：
- 原文完整保留 → 信息密度不丢
- AI 旁注帮助快速定位 → 用户决定看不看
- 标注"待读"段落 → 用户回来精读时有指引

**等于做了一个 AI 版的"荧光笔 + 便利贴"。**

**进阶：让用户选择。**
```
kb url --mode=fidelity   # 原文+旁注（默认）
kb url --mode=concise    # 去水提炼（用户主动选）
kb url --mode=raw        # 纯原文，不加工
```

**核心原则：默认不丢信息，用户主动选择要省略什么。**

**Boss 回复：好的** ✅

**决策：** 砍掉"去水留干"，改为**"原文+AI 旁注"**（默认模式）。

### 议题 2：B站/抖音视频总结——有没有人做？

**Boss 提问：** 我记得百度网盘有视频生成AI笔记的，而且做得非常好，截图＋笔记

**灵虾调研结果：**

**GitHub 搜索：B站相关项目**

| 项目 | Stars | 做什么 | 评价 |
|------|-------|--------|------|
| **bilichat** | 267 | QQ bot 插件，B站视频总结 | 是 bot 插件，不是独立工具 |
| **bilibili-summary** | 9 | AI 视频总结 + ASR | 有雏形，没维护 |
| **video-summarizer** | 13 | YouTube/B站/Twitter→Markdown | 功能简陋 |
| **chatGPT-Summary-Bilibili-To-Notion** | 1 | 字幕→ChatGPT→Notion | 只支持B站 |
| **astrbot_plugin_bilibili_summary** | 6 | B站视频字幕总结 | 又是 bot 插件 |

**抖音：基本为零。** 搜 douyin summary / tiktok summary，没有像样的开源项目。

**YouTube：** 有几个 0-star 项目，都是简单调 YouTube API 拿字幕 → GPT 总结，非常粗糙。

**为什么没人做好：**
1. 大部分项目是"调 API + 调 GPT"的一次性脚本，不是产品
2. 没人做**结构化学习笔记**（摘要+知识框架+时间戳+关键概念）
3. 没人做**多平台统一入口**
4. 没人把"视频笔记"作为**个人知识库管线的一环**

**说"没有人做"不完全准确，但"没有人做好"是对的。**

**百度网盘「AI 笔记」分析：**

百度网盘确实有这个功能——上传视频后自动生成：
- **关键帧截图**（自动抓取重要画面）
- **时间戳笔记**（对应时刻的知识点）
- **内容总结 + 提纲**
- 支持网盘内的讲座/课程视频

**但跟我们的区别：**

| 维度 | 百度网盘 AI 笔记 | 我们的方案 |
|------|----------------|-----------|
| **入口** | 必须上传到百度网盘 | 任意链接，不需要上传 |
| **平台** | 只处理网盘内视频 | B站/YouTube/抖音/任意链接 |
| **输出** | 网盘内查看，导出受限 | 本地 Markdown，完全可控 |
| **AI** | 闭源黑盒 | 可自选模型和 prompt |
| **开源** | ❌ 百度私有 | ✅ MIT |
| **与笔记软件联动** | 无 | 直接进 Obsidian/Logseq |

**百度网盘证明了"截图 + 笔记"这个产品形态是对的，用户确实需要。** 但它绑定网盘、不可编程、不开放。我们的差异化：做一个开放的、CLI 的、多平台的、Markdown 原生的"百度网盘 AI 笔记"平替。

**Boss 回复：好的** ✅

**决策：** 参考百度网盘 AI 笔记形态，做**字幕+关键帧截图+时间戳笔记**。

### 议题 3：组件模块化 + 本地/API 双通道

**Boss 提问：** 我看看，一些可选安装的组件 我们可以预留下载开关，用户可以选本地处理或者api处理

**灵虾设计：**

**安装层级：**
```
pip install kb-tool          # 核心（<50MB）：网页提取+路由
pip install kb-tool[ocr]     # +OCR（~500MB）
pip install kb-tool[video]   # +视频管线（~60MB）
pip install kb-tool[ai]      # +AI 旁注客户端
pip install kb-tool[all]     # 全部
```

**每个组件本地/API 双通道：**

| 组件 | 默认 | 本地方案 | API 方案 |
|------|------|---------|---------|
| OCR | 本地 | PaddleOCR（CPU，~500MB） | 百度/腾讯/Azure OCR API |
| LLM 旁注 | API | 本地模型 via Ollama（~4GB） | OpenRouter / OpenAI API |
| 视频字幕 | 本地 | yt-dlp + Whisper 本地（~3GB） | Whisper API / 讯飞 API |
| 网页提取 | 本地 | trafilatura（内置） | Jina Reader API / Firecrawl |

**未安装对应模块时，提示用户安装或切 API，不直接报错退出：**
```
$ kb "https://bilibili.com/video/BVxxx"
⚠️ 视频管线未安装。运行：
   pip install kb-tool[video]    # 本地处理（推荐）
   或使用 API：kb config set video.provider api
```

**Boss 回复：好的** ✅

**决策：** 模块化安装 + 每个组件支持本地/API 切换。

### 议题 4：LLM 要不要本地模型？

**Boss 提问：** llm没必要本地吧？openclaw接入，或者api即可，本地小模型能处理这样的业务吗

**灵虾分析：**

你说得对，砍掉。

**为什么本地 LLM 没必要：**

AI 旁注的任务其实很难：
- 要判断"这段的核心论点是什么"
- 要判断"这段值不值得回头精读"
- 要不改变原意、不遗漏关键信息

**7B 小模型做不好这种事。** 精度不够，旁注质量差 = 产品没价值。

**更好的方案：**

| LLM 来源 | 适合场景 | 成本 |
|----------|---------|------|
| **OpenClaw 接入**（推荐） | Boss 用的时候直接走现有 LLM | ¥0 额外成本 |
| **OpenRouter API** | CLI 独立用户 | ~¥0.03/篇 |
| ~~本地模型~~ | ~~砍掉~~ | ~~质量不达标~~ |

**LLM 走 OpenClaw 的优势：**
1. Boss 已经配好了模型，复用就行
2. 模型升级自动跟着走（不用单独维护）
3. 成本已经在 OpenClaw 里了
4. 质量有保障（用的是正经大模型不是 7B）

**这个产品定位从"全栈独立工具"变成"OpenClaw 生态的原生能力"更合理。** 很多用户本身就是 OpenClaw 用户，天然集成。

**Boss 回复：好的** ✅

**决策：** 不做本地 LLM，砍掉。LLM 走 OpenClaw / OpenRouter API。

## 竞品调研完整结论

| 竞品 | Stars/背景 | 覆盖领域 | 与我们的核心区别 |
|------|-----------|---------|----------------|
| **MarkItDown** | 微软出品 | 文件→Markdown（PDF/Word/PPT/图片/YouTube） | 只做格式转换，不做内容理解，没有"笔记"概念 |
| **Crawl4AI** | 50k+ star | 网页→LLM-friendly Markdown | 只做网页，面向 RAG 不面向笔记 |
| **Jina Reader** | Jina AI 出品 | URL→LLM 文本（API 服务） | 云端 API，非本地工具 |
| **ScrapeGraphAI** | 高 star | AI 驱动网页抓取 | 只做抓取，不做提炼和笔记 |
| **百度网盘 AI 笔记** | 百度出品 | 视频→笔记（截图+时间戳） | 绑定网盘，不可编程，不可导出 |
| **bilichat** | 267 star | B站视频总结 | QQ bot 插件，非独立工具 |
| **Obsidian Importer** | 官方插件 | 笔记格式迁移 | 只做导入，不做内容提取 |

**关键发现：**
1. **没人做"智能路由"** —— 所有工具都需要用户指定类型
2. **没人做"原文+AI 旁注"** —— 要么纯转换，要么提炼，没有中间态
3. **没人做"视频→结构化学习笔记"** —— B站/抖音完全空白
4. **百度网盘 AI 笔记形态被验证** —— 但绑定生态，不可编程
5. **微软 MarkItDown 是最大潜在威胁** —— 但只做转换不做理解，护城河在"旁注"而不是"转换"
